# RAI 제품 로드맵 - 헤드헌터 인터뷰 기반

## Executive Summary

헤드헌터 인터뷰 분석 결과, RAI의 현재 MVP(이력서 분석 + 검색 + JD 매칭)는 **"찾기"** 기능에 집중되어 있으나, 헤드헌터의 실제 수익 창출 포인트인 **"빠르게 + 정확히 + 적절한 타이밍에"** 연결하는 기능이 부족합니다.

### 핵심 인사이트
> "헤드헌팅에서 커미션은 보통 '먼저 좋은 후보자를 제출한 사람'이 먹는다"
> "검색이 좋아져도 성과가 안 오르는 케이스: DB가 썩어있거나, 누구부터 연락해야 할지 결정 못 함"

---

## Gap Analysis: 인터뷰 Pain Points vs 현재 서비스

| 헤드헌터 Pain Point | 현재 RAI 기능 | Gap |
|---------------------|---------------|-----|
| 좋은 후보자를 "지금" 찾아야 함 | ✅ 하이브리드 검색 | - |
| 후보자 이직 의향/상태 파악 | ❌ 없음 | **Critical** |
| 마지막 연락일 기준 재활성 | ❌ 없음 | **Critical** |
| Warm Pool 사전 세그먼트 | ⚠️ Projects 폴더만 | **High** |
| JD 받고 즉시 매칭 | ⚠️ 수동 새로고침 필요 | **Medium** |
| 제출 패키지 자동 생성 | ⚠️ Blind Export만 | **Medium** |
| 캠페인 스케줄링 | ❌ 없음 | **Low (Phase 2)** |

## 우선순위별 개선 기획안

### 🔴 P0: 후보자 라이프사이클 관리 (Candidate Lifecycle)

**문제**
- "좋은 사람은 이미 수십 명에게 연락 받는 중"
- "예전 후보자 DB 재활성이 돈 됨" → 근데 언제 연락했는지, 이직 의향이 뭔지 모름

**솔루션**
후보자 레코드에 CRM 스타일 필드 추가:

```
- last_contact_at: 마지막 연락 일시
- interest_level: hot/warm/cold/unknown
- salary_expectation_min/max: 희망 연봉 범위
- location_preference: 희망 근무 지역
- earliest_start_date: 최소 입사 가능일
- availability_notes: 제약조건 메모
```

**예상 임팩트**
| 지표 | Before | After | 개선율 |
|------|--------|-------|--------|
| 재활성 후보자 발굴 시간 | 2-4시간 | 15-30분 | **75-85%↓** |
| 중복 연락 실수 | 빈번 | 0 | **100%↓** |
| "지금 움직일 사람" 특정 | 감으로 | 필터로 | **정확도↑** |

**기술 리뷰 결과**
- 구현 복잡도: **Small**
- 기술 준비도: 4/5
- 예상 개발: 2-3일
- 의존성: 없음

---

### 🔴 P1-A: JD 업로드 → 자동 매칭 (Instant Match)

**문제**
- "TTR(Time To Recommend) 단축이 경쟁력"
- "JD 받고 오늘 안에 3-5명 제출이 가능해야"

**현재 흐름**
```
JD 업로드 → Position 생성 → 수동으로 매칭 새로고침 → 결과 확인
(4단계, 수동 개입 필요)
```

**개선 흐름**
```
JD 업로드 → 즉시 상위 20명 매칭 결과 + Position 자동 생성
(1단계, 원클릭)
```

**예상 임팩트**
| 지표 | Before | After | 개선율 |
|------|--------|-------|--------|
| JD→숏리스트 시간 | 2-4시간 | 10-30초 | **99%↓** |
| 일일 처리 가능 포지션 | 1-2개 | 5-10개 | **5x↑** |

**기술 리뷰 결과**
- 구현 복잡도: **Medium**
- 기술 준비도: 5/5 (기존 코드 조합)
- 예상 개발: 3-4일
- 의존성: 없음

**핵심 구현**: 기존 `/api/positions/extract` + `/api/positions/[id]/matches` 통합

---

### 🔴 P1-B: 재활성 검색 필터 (Reactivation Search)

**문제**
- "분기 1회 Warm Pool 상태 갱신 캠페인" 필요
- "마지막 대화에서 후보자가 말한 진짜 이직 동기" 기반 개인화

**솔루션**
검색 필터 확장:

```typescript
// 새로운 필터
{
  lastContactBefore: "2024-10-01",   // 3개월 이상 미접촉
  lastContactAfter: "2024-01-01",    // 최근 1년 내 접촉한 적 있음
  interestLevel: ["warm", "hot"],    // 이직 의향 있는 사람만
  excludeRejected: true,             // 탈락 이력 제외
  notInPosition: "uuid-xxx"          // 특정 포지션 지원자 제외
}
```

**예상 임팩트**
| 지표 | Before | After | 개선율 |
|------|--------|-------|--------|
| Warm Pool 구성 시간 | 수작업 | 필터 1번 | **95%↓** |
| 재활성 응답률 | 낮음 (무작위) | 높음 (타겟) | **3-5x↑** |

**기술 리뷰 결과**
- 구현 복잡도: **Small**
- 기술 준비도: 4/5
- 예상 개발: 1-2일
- 의존성: **P0 (Lifecycle) 필수 선행**

---

### 🟡 P2: 후보자 제출 패키지 (Candidate Packaging)

**문제**
- "후보자 매칭만 해주면, 결국 헤드헌터가 다시 문서 만듦"
- "JD 기준 매칭 근거 3-5줄 + 리스크 + 확인 질문이 바로 뽑혀야"

**현재**
- Blind Export: PII 제거된 후보자 기본 정보만

**개선**
- JD 컨텍스트 기반 패키지:

```
📋 후보자 제출 패키지
─────────────────────────
[기본 정보] 이름, 경력, 스킬 (기존 Blind Export)

[매칭 분석] ← NEW
✅ 매칭 이유 (3-5개 bullet)
- 결제 도메인 5년 경력 → JD 요구사항 충족
- Kafka/Spark 실무 경험 → 데이터 파이프라인 역량
- B2B SaaS 스케일업 경험 → 성장 단계 핏

⚠️ 잠재 리스크 (2-3개)
- 리더십 경험 부족 → 팀 리드 역할 검증 필요
- 최근 이직 주기 짧음 → 동기 확인 필요

[면접 질문 제안] ← NEW
1. Kafka 메시지 유실 처리 경험
2. 결제 장애 대응 사례
3. 팀 갈등 해결 경험
```

**예상 임팩트**
| 지표 | Before | After | 개선율 |
|------|--------|-------|--------|
| 제출 문서 작성 시간 | 30-60분/명 | 2분/명 | **95%↓** |
| 제출 퀄리티 | 일관성 없음 | 표준화 | 재의뢰↑ |

**기술 리뷰 결과**
- 구현 복잡도: **Medium**
- 기술 준비도: 5/5
- 예상 개발: 3-4일
- GPT-4 호출 비용: 패키지당 ~$0.05

---

### 🟢 P3: 캠페인/풀 관리 (Campaign Management)

**문제**
- "캠페인 단위로 리스트를 뽑아야"
- "1차/2차/3차 접촉 로그가 자동으로 누적"
- "중복 컨택으로 욕먹는 걸 막아야"

**솔루션**
Projects를 Campaign으로 확장:

```
Campaign = {
  name: "Fintech Data Eng Warm Pool",
  filter_criteria: { skills: ["Spark", "Kafka"], interest: "warm" },
  outreach_schedule: { frequency: "weekly" },
  candidates: [
    { id: "xxx", outreach_status: "pending", outreach_count: 0 },
    { id: "yyy", outreach_status: "contacted", outreach_count: 2 }
  ]
}
```

**예상 임팩트**
- "운"이 아니라 "시스템"으로 전환
- DB가 자산화됨 (파일 창고 → 파이프라인)

**기술 리뷰 결과**
- 구현 복잡도: **Large**
- 기술 준비도: 3/5
- 예상 개발: 1-2주
- 의존성: P0, P1-B 선행 권장

## 매출 영향 분석 (헤드헌터 인터뷰 기반)

> "헤드헌터에게 시간 절약은 좋은데, 진짜 달콤한 건 커미션 건수 1건 증가"

### 매출 공식
```
매출 = (유효 제출 수) × (면접 전환율) × (오퍼 전환율) × (입사 전환율) × (커미션 단가)
```

### RAI 개선이 영향을 미치는 변수

| 기능 | 영향 변수 | 예상 개선 |
|------|-----------|-----------|
| JD Auto-Match | 유효 제출 수 ↑ | 3→6명/포지션 |
| Reactivation Search | 유효 제출 수 ↑ | 숨겨진 후보 발굴 |
| Candidate Packaging | 면접 전환율 ↑ | 미스매치 감소 |
| Lifecycle Management | 입사 전환율 ↑ | 이탈/번복 사전 감지 |

### 시나리오 분석
```
Before (월간):
- 포지션 3개 × 제출 3명 = 9명 제출
- 면접 3명 → 오퍼 1개 → 입사 0.8개
- 커미션: 0.8 × 1000만원 = 800만원

After (월간):
- 포지션 5개 × 제출 5명 = 25명 제출 (JD Auto-Match)
- 면접 8명 → 오퍼 3개 → 입사 2개 (품질 향상)
- 커미션: 2 × 1000만원 = 2000만원

예상 증가: +150% (포지션 커버리지 + 전환율)
```

## 기술 리뷰 요약 (시니어 엔지니어)

### 구현 준비도
- 기존 코드베이스가 잘 구조화되어 있어 확장 용이
- 검색 RPC 함수 패턴이 이미 확립됨
- GPT-4 통합 파이프라인 성숙

### 주요 리스크
1. **P0**: DB 마이그레이션 필요 (안전, 하위 호환)
2. **P1-A**: 동기 처리 시 5초 이내 응답 필요 (성능 최적화)
3. **P2**: GPT-4 비용 관리 (캐싱 전략 필수)
4. **P3**: UI 작업량 큼 (캠페인 빌더)

### 권장 순서
```
P0 (Foundation) → P1-A/P1-B (Quick Wins) → P2 (Value Add) → P3 (Long-term)
```
